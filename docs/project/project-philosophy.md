# Project philosophy

Enterprise Architecture Academy exists to teach people how to think and work as Enterprise Architects.

Certification is an intended outcome, but it is not the purpose of the project. The curriculum must help learners frame enterprise problems, identify stakeholders and concerns, make trade-offs explicit, define useful current and target states, plan transitions, and govern change.

## The curriculum is a product

The repository is treated as an engineered educational product rather than a folder of notes.

- Important decisions are documented.
- Content follows shared structures and standards.
- Changes are reviewed through pull requests.
- Published outputs are generated from a controlled source.
- The curriculum is revised when evidence, standards, learner results, or certification requirements change.

## One editorial vision

Community contribution is encouraged, but consensus is not the editorial model. The repository owner retains final authority over scope, sequence, tone, acceptance, and publication.

This preserves coherence while still allowing corrections, examples, exercises, and new material from others.

## Teach the work, not only the vocabulary

Framework terminology is taught in context. Lessons begin with recognizable enterprise problems and introduce framework concepts as tools for solving them.

A learner who finishes the curriculum should be able to explain not only what a concept means, but when it is useful, what decision it supports, what evidence it requires, and what can go wrong when it is misapplied.

## Persistent scenarios

The curriculum reuses a small number of fictional enterprises. Their environments, constraints, risks, and decisions evolve throughout the course.

Persistent scenarios allow learners to see how architecture work accumulates over time rather than treating each lesson as an isolated exercise.

## Canonical source and generated outputs

Markdown stored in Git is the authoritative source. The MkDocs website, PDF releases, DOCX exports, slide decks, and other formats are derived outputs.

Parallel manual versions are avoided because they create drift, conflicting revisions, and uncertainty about which copy is authoritative.

## Definition of success

The project succeeds when learners can:

- recognize enterprise-level problems;
- distinguish architecture from engineering, operations, and project management;
- apply framework concepts without treating the framework as a rigid script;
- make defensible recommendations using evidence and explicit trade-offs;
- communicate architecture to technical and nontechnical stakeholders;
- pass the relevant certification examinations without relying on recalled exam content.
